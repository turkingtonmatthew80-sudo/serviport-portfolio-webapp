**Versión:** 2.0 (Integración de Motor de Simulación Autónoma y Operaciones Reales)
**Autor:** Arquitectura Principal de Sistemas
**Entorno:** Nodos Portuarios Venezolanos (Puerto Cabello, La Guaira, Maracaibo, Guanta)

---

## 1. VISIÓN GLOBAL DEL SISTEMA (EL GEMELO DIGITAL HÍBRIDO)

**Serviport OS** no es solo un ERP/TOS tradicional. Es un **Gemelo Digital Híbrido**. Funciona bajo dos motores paralelos que convergen en una única interfaz visual (2D/3D) y de datos:

1.  **Motor de Simulación Autónoma (Background):** Un sistema alimentado por algoritmos de *Scraping* que ingiere datos de tráfico marítimo público (AIS, APIs de rastreo). Simula de manera algorítmica y discreta las operaciones del puerto (atraque, grúas, descarga, distribución de patio) para todos los buques ajenos a la empresa. Su propósito es estético, analítico (para entender la congestión real del puerto) y de marketing (dashboards vivos en la landing page).
2.  **Motor de Operaciones Reales (User-Driven):** El ERP/SaaS transaccional. Aquí es donde los clientes de Serviport (Armadores, Navieras) y los empleados (Gerencia, Water Clerks, Yard Planners) interactúan. Las operaciones registradas aquí generan facturación, afectan el inventario real y se integran legalmente con aduanas (SIDUNEA / INEA).

**La Convergencia:** En los módulos visuales (TOS 2D/3D), el usuario ve el puerto operando al 100%. El 90% de los barcos y grúas moviéndose son controlados por la *Simulación Autónoma*, pero el 10% son los buques de los clientes de Serviport, controlados por el *Motor de Operaciones Reales*.

## 2. ARQUITECTURA DE DATOS: SCRAPING Y SIMULACIÓN DISCRETA

### 2.1. El Motor de Inteligencia Marina (Scraper)
* **Tecnología:** Celery Beat Tasks (Python) consumiendo APIs como Apify (Marine Traffic Scraper), Firecrawl y AISStream.
* **Frecuencia:** Cada 15 minutos.
* **Target:** Nodos `VE PCL` (Puerto Cabello), `VE LGU` (La Guaira), `VE MAR` (Maracaibo), `VE GUA` (Guanta).
* **Data Ingerida:** IMO del buque, Nombre, Eslora, Calado, Velocidad, ETA, Tipo de Buque (Container, Tanker, Bulk).

### 2.2. Base de Datos de Enriquecimiento (Static DB)
Al recibir un IMO scrapeado, el sistema consulta la tabla `vessels_registry`. 
* Si el buque existe, extrae sus planos 3D aproximados, capacidad de TEUs y naviera propietaria.
* Si no existe, crea un registro base genérico y asigna un modelo 3D según su categoría (ej. Panamax Genérico).
* También se nutre de la tabla `port_infrastructure` (coordenadas reales de los 25 puestos de atraque de Puerto Cabello, silos, almacenes del SENIAT).

### 2.3. Simulación de Eventos Discretos (DES)
No dependemos de usuarios para que el puerto "tenga vida". El backend en FastAPI ejecuta un loop de simulación espacial y temporal:
1.  **Atraque Algorítmico:** El buque scrapeado con ETA cercano es asignado a un puesto de atraque (Berth) *no reservado por Serviport*.
2.  **Generación de Carga Ficticia:** Basado en el calado del buque, el algoritmo calcula cuántos TEUs trae. Genera una lista en caché (Redis) de contenedores con números ISO 6346 ficticios pero válidos.
3.  **Animación Operativa:** El sistema emite eventos por WebSocket (Pub/Sub en Redis) que indican: `{"crane_1": "moving", "container_id": "HLXU1234567", "from": "vessel", "to": "truck_4"}`.
4.  **Distribución Matemática:** Se aplican algoritmos de optimización de TOS modernos (ej. *Heurística de apilamiento*) para decidir a qué bloque del patio simulado va cada contenedor ficticio.

> **Regla de Aislamiento:** Los datos generados por la simulación viven **únicamente en Redis** (para broadcast de WebSockets) y en una base de datos de series de tiempo (ClickHouse o TimescaleDB) para los Dashboards. **NUNCA** se insertan en las tablas relacionales de facturación o inventario de PostgreSQL.

## 3. FLUJO DE USUARIOS E INTERFACES (MOTOR DE OPERACIONES REALES)

El sistema transaccional (PostgreSQL) gobierna sobre la simulación. Cuando un usuario humano actúa, el sistema le otorga máxima prioridad.

### 3.1. Clientes B2B (Armadores, Agencias de Carga, Importadores)
* **Interface:** Portal SaaS (diseño minimalista, alto contraste, modo oscuro/claro).
* **Permisos:** Crear solicitudes (Port Calls), ver facturas (Proformas), rastrear **solo** su carga.
* **Flujo de Acción:**
    1. El cliente ingresa al portal y crea un **Port Call Request** para el buque "CMA CGM PEMBA" (IMO: 9389332).
    2. Sube archivos EDIFACT (BAPLIE, COARRI) o PDFs.
    3. *Convergencia:* El algoritmo de simulación detecta que este IMO ya no es "Autónomo". Lo retira del pool de simulación y le asigna el estatus de `REAL_OPERATION`. El puesto de atraque asignado se bloquea en la maqueta 3D para otras simulaciones.
    4. El cliente aprueba la Factura Proforma generada por el Tariff Engine (Motor de Tarifas).

### 3.2. Gerente de Operaciones (SuperUser)
* **Interface:** Command Center Dashboard. Gráficos densos (AG Grid Enterprise).
* **Permisos:** Visión panóptica. Aprueba presupuestos, altera tarifas, ve márgenes de ganancia.
* **Flujo de Acción:**
    1. Recibe la alerta del Port Call creado por el cliente B2B.
    2. Revisa el cruce con INEA/Bolipuertos y aprueba la operación (`status: APPROVED`).
    3. Visualiza un Dashboard mixto: "Tráfico del Puerto (Simulado)" vs "Operaciones Propias (Reales)".

### 3.3. Water Clerk (Operador de Muelle)
* **Interface:** PWA (Progressive Web App) en Tablet/Móvil. *Debe tener Offline Mode (Service Workers + IndexedDB)* debido a la mala conexión 4G en muelles venezolanos. Sincroniza al recuperar señal.
* **Permisos:** Actualizar estado de buques físicos, registrar incidencias de Husbandry.
* **Flujo de Acción:**
    1. Se para frente al buque físico en el muelle de Puerto Cabello.
    2. Registra "Líneas Aseguradas" (Atraque finalizado).
    3. Registra servicios auxiliares: "Suministro de Agua Potable" o "Cambio de Tripulación".
    4. Cada toque de botón hace un `POST /api/v1/operations/milestone`. Esto dispara un Webhook al ERP financiero para sumar el costo a la factura final.

### 3.4. Yard Planner (Planificador de Patio)
* **Interface:** Monitor Ultra-Wide. Pantalla dividida. Izquierda: Matriz de datos (AG Grid). Derecha: TOS 2D (Konva.js).
* **Permisos:** Drag & Drop de contenedores, asignación de cuadrillas.
* **Flujo de Acción:**
    1. El Yard Planner ve la proyección de descarga del BAPLIE (parseado por Celery en el backend).
    2. Asigna en el mapa 2D las coordenadas exactas `Block A, Row 5, Tier 3` para contenedores Reefer (Refrigerados).
    3. *Convergencia:* En su pantalla, los contenedores "Simulados" (grises) ocupan espacio para que pueda ver la congestión real del puerto, pero sus contenedores "Reales" (naranjas/azules según naviera) son los únicos interactivos.

### 3.5. Gate Checker (Control de Acceso)
* **Interface:** PWA modo "Kiosko" en el punto de control de Bolipuertos/Aduana. Lector de QR/Barcode usando la cámara del dispositivo.
* **Flujo de Acción:**
    1. Llega un camión gandola. El chófer muestra su EIR (Equipment Interchange Receipt).
    2. El Gate Checker escanea. El sistema hace match con el `booking_id` del cliente.
    3. Registra `Gate-In`. Esto avisa al Yard Planner y actualiza el estado del contenedor a `IN_YARD`.

## 4. FLUJO DE DATOS Y CONEXIONES DEL SISTEMA (TOPOLOGÍA TÉCNICA)

El flujo de información debe garantizar que las acciones manuales actualicen la simulación matemática sin latencia perceptible.

### 4.1. Arquitectura de Eventos (Pub/Sub)
1. **Frontend (Next.js):** Mantiene un estado global usando **Zustand**. Está suscrito a un WebSocket `wss://api.serviport.com/ws/live-port`.
2. **Backend (FastAPI):** * Tiene un proceso en *Background* (`asyncio.create_task`) ejecutando la Simulación Matemática a 10 Ticks por segundo. Este proceso publica JSONs en un canal de Redis `channel:port_simulation`.
   * Tiene Endpoints REST (`/api/v1/...`) para las operaciones transaccionales.
3. **Flujo de una Operación Real:**
   * El *Yard Planner* mueve un contenedor en la UI.
   * Frontend envía `POST /api/v1/tos/moves {"container_id": "X", "target": "A-5-3"}`.
   * FastAPI valida ACID rules en PostgreSQL. Si es exitoso, guarda en DB.
   * FastAPI dispara un evento a Redis: `PUBLISH port_real_ops {"type": "MOVE", "data": ...}`.
   * El WebSocket propaga el evento.
   * La UI de todos los usuarios conectados (Clientes, Gerente) se actualiza instantáneamente; el contenedor vuela en el modelo 3D/2D de una posición a otra.

### 4.2. Integración Documental (Legal & Aduanas)
El sistema debe integrarse teóricamente con la realidad legal venezolana.
* **Módulo SIDUNEA:** Cuando un *Gate-Out* ocurre, Celery toma los datos de la carga, la nacionalización y el RIF del importador, y genera un PDF idéntico al formato DUA (Documento Único Aduanero) del SENIAT.
* **Facturación Electrónica:** Al cerrar un Port Call, el motor de ERP consolida los hitos del *Water Clerk* y los movimientos del *Yard Planner*, aplica las "Tarifas Vigentes" de Serviport, y emite una factura con cálculo automático de IVA y retenciones en Bolívares (referenciados a Tasa BCV API) y Dólares.

## 5. DICCIONARIO DE DATOS Y ESTRUCTURAS CRÍTICAS (JSONB & EDI)

Debido a que el negocio marítimo maneja estándares de los años 80 (EDIFACT / ANSI X12), el sistema debe modernizar esto usando `JSONB` en PostgreSQL para permitir metadatos dinámicos.

### 5.1. Esquema Híbrido de Buques (Vessels)
```sql
CREATE TABLE vessels (
    id UUID PRIMARY KEY,
    imo_number VARCHAR(7) UNIQUE NOT NULL,
    name VARCHAR(100) NOT NULL,
    type VARCHAR(50), -- CONTAINER, RO-RO, BULK
    dimensions JSONB, -- {"length": 294, "beam": 32, "draft": 11.5}
    is_simulated BOOLEAN DEFAULT TRUE, -- TRUE = Scrapeado, FALSE = Cliente Real
    last_known_position JSONB, -- {"lat": 10.48, "lon": -68.01, "timestamp": "..."}
    created_at TIMESTAMP DEFAULT NOW()
